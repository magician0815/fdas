## 开发策略

### 开发方式选择规则

根据模块类型自动选择开发方式，无需用户指令：

| 模块类型 | 开发方式 | 原因 |
|----------|----------|------|
| SQLAlchemy模型定义 | **直接开发** | 纯数据结构，无复杂逻辑 |
| 配置/日志验证 | **直接开发** | 基础设施代码 |
| 密码服务(bcrypt) | **直接开发** | 简单封装 |
| 用户CRUD API | **/tdd** | 业务逻辑，需要验证 |
| Session管理 | **/tdd** | 认证流程，需要验证 |
| 数据采集服务 | **/tdd** | 复杂流程，需要重试逻辑 |
| 技术指标计算(MA/MACD) | **/tdd** | 数学计算，需要精度验证 |
| APScheduler配置 | **直接开发** | 基础设施代码 |
| 前端组件 | **直接开发** | 后续用E2E测试验证 |
| 集成测试 | **直接开发** | 验证整体流程 |

**执行规则**：
- 直接开发：完成代码后立即使用 `/code-review` 检查
- TDD开发：使用 `/tdd` 命令启动，确保RED→GREEN→IMPROVE流程

### 最小功能增量原则

每次只完成**1个模块**，完成后再进行下一个，避免批量开发导致的复杂性和回滚困难。

## 版本管理规范

### Git提交规则

每个模块开发完成后，自动执行以下流程：

1. **代码审查** - 使用 `/code-review` 检查代码质量
2. **提交代码** - 按规范格式提交
3. **推送到GitHub** - 同步远程仓库

### 提交消息格式

```
feat(module): 模块名称简述

完成内容:
- 具体内容1
- 具体内容2

验收:
- 验收标准1 ✅
- 验收标准2 ✅
```

### 模块完成触发器

每完成一个模块后，自动执行：
```bash
git add .
git commit -m "feat(module): xxx"
git push origin main
```
